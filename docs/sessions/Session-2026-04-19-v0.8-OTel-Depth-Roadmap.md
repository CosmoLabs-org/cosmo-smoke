## Summary

This session implemented four OpenTelemetry depth roadmap items (ROAD-035, ROAD-036, ROAD-037, ROAD-038) that expand the OTel trace correlation feature shipped in Session 004 (v0.8.0). All four items were executed sequentially on master, adding 526 lines of new Go code across 5 new files with 25 new tests. No new external dependencies were introduced.

The session tackled the four roadmap items in order. ROAD-037 (trace-aware retry) added `retry_on_trace_only` to `RetryPolicy`, enabling tests to skip retry cycles when the `otel_trace` assertion has already confirmed the trace was received by the backend -- saving time on flaky network paths where the trace lands but the assertion timing is tight. ROAD-036 (multi-backend trace verification) extended the `otel_trace` assertion beyond Jaeger to support Tempo, Honeycomb, and Datadog backends via a `traceBackend` interface, each with its own API path and response parsing logic. ROAD-035 (OTLP telemetry export) created an `OTelReporter` that emits OTLP JSON spans to an OpenTelemetry collector, using manual JSON construction (no SDK dependency) and a `MultiReporter` fan-out pattern that allows terminal and OTLP reporters to run simultaneously. ROAD-038 (watch mode trace health monitoring) introduced a `TraceHealthTracker` with a sliding window of the last 10 runs, displaying trace health percentage in the terminal summary and alerting when health drops below 50%.

Housekeeping included updating IDEA-MO1FC22M (mobile deep link assertion) to reflect multi-project-type scope.

## Key Decisions

| Decision | Options Considered | Why This Choice |
|----------|-------------------|-----------------|
| `traceBackend` interface for multi-backend | Switch/case on backend name, interface + registry, interface with constructor | Interface with constructor keeps each backend self-contained and testable; avoids a global registry that would grow unchecked |
| Manual OTLP JSON construction vs. OTel Go SDK | Add OTel Go SDK, manual JSON, protobuf marshal | Manual JSON stays consistent with cosmo-smoke's zero-new-deps philosophy; OTLP JSON protocol is simple enough for the limited span data needed |
| MultiReporter fan-out vs. replace | Replace terminal reporter, fan-out both, conditional selection | Fan-out allows teams to see terminal output while also shipping telemetry; selecting one would lose either UX or observability |
| Sliding window of 10 for trace health | 5, 10, 20, configurable | 10 runs provides enough signal without excessive memory; matches typical watch-mode session length; configurable can be added later |
| 50% alert threshold for trace health | 30%, 50%, 75%, configurable | 50% is the point where trace delivery is worse than a coin flip -- clearly actionable; configurable threshold left for future refinement |
