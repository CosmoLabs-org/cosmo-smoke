# Session 004 - 2026-04-19

## Date
2026-04-19

## Branch
master

## Summary

This session implemented the full OpenTelemetry trace correlation feature (FEAT-012, sourced from IDEA-MO1FBPJB), taking it from idea triage through design, planning, and complete implementation in a single session. The feature enables cosmo-smoke to generate W3C traceparent headers, propagate them into HTTP/WebSocket/gRPC assertions, and verify trace data landed in a Jaeger backend -- giving teams end-to-end observability of their smoke test runs without adding any external dependencies to the binary.

The session began with idea triage. IDEA-MO1FBPJB (OpenTelemetry trace correlation) was selected from the backlog based on portfolio-wide applicability: with ~95 projects at CosmoLabs, trace correlation in smoke tests fills a real gap between "tests pass" and "we can see what happened." The design was worked out through interactive Q&A, arriving at three core design choices: (1) a global `otel` config block in `.smoke.yaml` for collector endpoint and service name, (2) W3C traceparent propagation into HTTP, WebSocket, and gRPC requests, and (3) a `trace_check` assertion type that polls the Jaeger API to verify traces were received. The design spec was written to `docs/brainstorming/2026-04-18-opentelemetry-trace-correlation.md`, reviewed (6 issues found, all fixed), and then promoted to an implementation plan in `docs/planning-mode/2026-04-19-opentelemetry-trace-correlation.md` with 10 tasks organized into 6 parallelizable chunks.

All 10 tasks were executed sequentially on master. The schema layer gained `OTelConfig` (global collector address, service name, sampling) and `OTelTraceCheck` (Jaeger API polling with configurable timeout/retries), plus full validation including collector URL format checks and mutual exclusivity rules. The runner layer gained `TraceContext` and `SpanContext` structs for W3C traceparent generation (`trace.go`, 49 lines), `CheckOTelTrace` for Jaeger API verification (`assertion_otel.go`, 102 lines), and trace header injection into the three network-capable assertions: HTTP (`assertion_network.go`), WebSocket (`assertion_ws.go`), and gRPC (`assertion_grpc.go` + `assertion_grpc_stub.go`). The runner pipeline was wired to initialize trace context per-test when OTel is configured, carry it through assertion execution, and clean up. Two CLI flags were added to `cmd/run.go`: `--otel-collector` for ad-hoc endpoint override and `--no-otel` to disable OTel even when configured.

The implementation added 356 lines of new Go code across 4 files (`trace.go`, `trace_test.go`, `assertion_otel.go`, `assertion_otel_test.go`) with 16 new test cases. The test suite grew from 253 to 269 tests, all passing. Self-smoke remains 6/6. CLAUDE.md was updated to reflect 27 total assertion types (up from 26). The feature required zero new external dependencies -- Jaeger polling uses stdlib `net/http`, and the W3C traceparent format is pure string generation.

Housekeeping included creating FEAT-012 in the issue tracker and filing 5 roadmap items (ROAD-034 through ROAD-038) covering OTel feature variants and future integrations.

## Key Decisions

| Decision | Options Considered | Why This Choice |
|----------|-------------------|-----------------|
| Global `otel` config block vs. per-test config | Per-test only, global only, global + per-test override | Global config eliminates repetition; per-test override adds complexity without clear use case since service name and collector are project-level concerns |
| W3C traceparent format | W3C traceparent, Jaeger Uber trace ID, B3 | W3C is the open standard; Jaeger supports it natively; B3 is Zipkin-specific and declining |
| Jaeger API polling for trace verification | Jaeger API, OTLP export, both | Jaeger API is query-based and fits the "assert after run" pattern; OTLP export adds a dependency and is a push model that doesn't naturally fit assertion semantics |
| Zero new dependencies | Add OpenTelemetry Go SDK vs. stdlib-only | Staying consistent with cosmo-smoke's minimal-dep philosophy; the feature only needs traceparent generation (string formatting) and HTTP polling, not a full OTel SDK |
| `--no-otel` CLI flag | No flag, env var only, CLI flag | CLI flag is discoverable and explicit; env var is implicit and easy to miss; both is unnecessary complexity |
