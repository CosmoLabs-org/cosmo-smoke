## Strategic Priorities (Ordered)

### Priority 1: ROAD-032 — Claude Desktop MCP Extension

**Scope**: Large. Needs its own design session before implementation.

The big strategic pivot. Makes cosmo-smoke usable from Claude Desktop via MCP protocol. Users can ask Claude to generate, run, and debug smoke tests conversationally.

**What to figure out first**:
- MCP tool definitions: which smoke operations to expose (run, init, validate, list-tests)
- How to handle long-running test execution in MCP context (streaming vs polling)
- Config discovery: how Claude Desktop finds `.smoke.yaml` files in a project
- Error surfacing: assertion failures formatted for conversational debugging
- Whether this is a standalone MCP server binary or a mode of the existing `smoke` binary

**Recommended approach**: Design session first (brainstorm + plan), then implement as `smoke mcp` subcommand or separate binary.

### Priority 2: ROAD-039 — Portfolio Smoke Dashboard

**Scope**: Medium-large.

Dashboard that aggregates `smoke run --format json` results across the ~95-project CosmoLabs portfolio. Uses `smoke serve` endpoint to provide real-time status.

**Key design questions**:
- Backend: in-process HTTP server in `smoke serve` or separate dashboard binary
- Data collection: pull model (dashboard fetches from each project) vs push (projects report to dashboard)
- Storage: in-memory ring buffer vs persistent store
- Auth: portfolio-wide API key or per-project tokens

### Priority 3: ROAD-010 — Monorepo Sub-Config Support

**Scope**: Medium. Has existing plan.

`--monorepo` flag auto-discovers `.smoke.yaml` in subdirectories. Unlimited depth, configurable exclusions. Already has architectural design in place.

**Carry-over**: Check `docs/planning-mode/` for existing plan file.

### Priority 4: ROAD-033 — Parallel-Agent Merge-Conflict SOP

**Scope**: Small. Process improvement.

SOP for handling merge conflicts when multiple GLM agents modify overlapping files. Documentation-only.
