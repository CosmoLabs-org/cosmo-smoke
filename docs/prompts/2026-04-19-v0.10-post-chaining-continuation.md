### Direction 2: Mobile Deep Link Assertions (IDEA-MO1FC22M)

**Scope**: Medium-large. New assertion type family.

The only seed idea in the backlog. Would add mobile-specific assertion types:

- Deep link resolution (Android Intent / iOS URL scheme)
- App launch verification via deep link
- Cross-platform support (Android, iOS, React Native)

**Prerequisite**: Design session to scope platform-specific tooling requirements. Mobile testing may need emulators or device farms, which changes the zero-deps philosophy.

### Direction 3: CI/CD Integration Docs

**Scope**: Small-medium. Documentation + templates.

Reusable CI/CD templates for common platforms:

- GitHub Actions workflow (`.github/workflows/smoke.yml`)
- GitLab CI template
- Generic Docker-based CI template
- Exit code semantics for CI gates
- `--format junit` for CI result ingestion
- `--report-url` for centralized result collection

**Why**: The tool works great locally but has no published CI integration patterns. For a 95-project portfolio, standardized CI templates are high leverage.

### Direction 4: Performance Baselines

**Scope**: Medium. New feature + persistence layer.

`smoke run --baseline` for persistent timing comparisons:

- Store timing data from `response_time_ms` and overall test duration
- Compare current run against historical baseline
- Flag regressions when tests exceed N% of baseline
- Persistent storage (JSON file or SQLite, reusing dashboard's store)

**Key design question**: Is this a new assertion type, a report mode, or a standalone subcommand?

### Direction 5: Watch Mode File Reporter Fix

**Scope**: Small. Bug fix.

Watch mode accumulates state across re-runs. The file-based reporter (and potentially others) may grow unbounded as files are re-read and results accumulate on each watch cycle. Needs investigation and fix.

**Investigation needed**: Confirm the exact accumulation mechanism. Is it in the reporter layer, the runner layer, or the watch orchestration?
